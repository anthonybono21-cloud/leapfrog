## Pre-QA Bugs (found during manual testing)

### BUG-001: Tiling not producing clean grid
- **Severity:** P1
- **What:** Windows overlap instead of tiling into non-overlapping grid
- **Expected:** 3x2 grid with no overlap on a standard Mac display
- **Actual:** Windows stack/overlap, not properly positioned
- **Screenshot:** ~/Desktop/Screenshot 2026-04-04 at 10.11.50 PM.png
- **Likely cause:** tile manager not accounting for macOS menu bar, dock, or display scaling (Retina)

### BUG-002: CNN custom consent not auto-dismissed
- **Severity:** P2
- **What:** CNN's "Legal Terms and Privacy" modal uses styled-components, not a standard cookie framework
- **Actual:** `LEAP_AUTO_CONSENT` didn't catch it; required `page.evaluate` to click
- **Fix idea:** Add text-matching fallback for "Agree" buttons in modal dialogs (not just cookie banners)

### BUG-005: Chime has no visual source indicator
- **Severity:** P1
- **What:** When sound chime fires for intervention, user can't tell WHICH tiled window triggered it
- **User report:** "I hear the chime but can't tell what window is chiming"
- **Fix idea:** Flash the HUD border red, or pulse the window title bar, synchronized with the chime

### BUG-006: @..@ overlay not visible at tile sizes
- **Severity:** P1
- **What:** The intervention overlay is inside the viewport, but at tiled window sizes (~400x350) it's too small/cropped to notice
- **User report:** "Can't see the leapfrog @..@ to know there's an issue"
- **Fix idea:** Add a prominent RED banner/flag on TOP of the browser window — visible at ANY tile size

### IDEA-002: Red flag label for intervention at tile scale
- **Priority:** v0.6.1 (HIGH)
- **What:** When intervention is detected, show a bold red indicator at the very top of the window that's visible even at small tile sizes — like a persistent top bar reading "NEEDS HUMAN" or a red `@..@`
- **Design:** Similar to Chrome's "Chrome is being controlled by automated test software" infobar — always visible, impossible to miss at any size

### IDEA-001: Adaptive viewport / zoom-to-fit when tiling
- **Priority:** v0.6.1
- **What:** When tiling, viewport stays 1280x720 but window tile is ~400x350, so user sees 1/4 of the page at full scale
- **Proposal:** Set viewport to match tile dimensions, OR inject `document.body.style.zoom` to scale content down
- **Benefit:** User sees full page in each tile instead of a cropped sliver

## Agent 2: Intervention & Consent

### BUG-007 (P1): Invisible reCAPTCHA triggers false-positive intervention overlay on nearly every major site
- **Severity:** P1 (blocks normal usage)
- **What:** The intervention detection in `src/intervention.ts` matches `iframe[src*="recaptcha"]` which catches Google's invisible reCAPTCHA (`recaptcha/api2/aframe`). This iframe is loaded by Google Ads / reCAPTCHA v3 on virtually every site with ads, even when no visible CAPTCHA challenge is presented.
- **Affected sites in this test:** bbc.com, lemonde.fr, techcrunch.com, reddit.com, nytimes.com, theguardian.com (6/6 sites tested)
- **Expected:** Overlay should only appear for visible, user-facing CAPTCHA challenges
- **Actual:** `@..@` overlay with "CAPTCHA detected" appears on every site that loads Google ads or reCAPTCHA v3 scoring, even though the page content is fully accessible behind the overlay
- **Root cause:** `CAPTCHA_IFRAME_PATTERNS` in `intervention.ts` line 36 includes `'recaptcha'` which matches `recaptcha/api2/aframe` (invisible scoring iframe). The `checkCaptchaIframes()` function at line 68 checks only the `src` attribute, not whether the iframe is visible or whether a challenge is actually presented.
- **Fix idea:** Add visibility check: only trigger if the reCAPTCHA iframe is visible (`offsetWidth > 0 && offsetHeight > 0`) AND/OR exclude `recaptcha/api2/aframe` (scoring-only) and only match `recaptcha/api2/anchor` or `recaptcha/api2/bframe` (actual challenge iframes). Also consider checking for `size=invisible` in the anchor URL parameters.
- **Screenshots:** `qa-consent-bbc.png`, `qa-consent-lemonde.png`, `qa-consent-techcrunch.png`, `qa-consent-reddit.png`

### Bugs Found

**BUG-005 (P2): Domain knowledge loaded but never used for navigation decisions**
- `src/index.ts` line 528: `const domainRecord = await domainKnowledge.load(urlDomain)` loads the record
- The variable `domainRecord` is NEVER passed to `adaptiveNavigate()` or used anywhere
- `adaptiveNavigate()` in `src/adaptive-wait.ts` does NOT accept a domain record parameter
- Result: learned wait strategies and stealth tiers are recorded but never consulted on revisits
- **Impact**: This is the core gap — the system "remembers" but does not "act on memories"

**BUG-006 (P3): Domain normalization inconsistency**
- `wikipedia.org` stores without `www.` prefix (navigated to `https://wikipedia.org`, stored as `wikipedia.org`)
- `www.espn.com`, `www.amazon.com`, `www.reddit.com` all keep the `www.` prefix
- Lookup with `www.wikipedia.org` returns "No knowledge" even though `wikipedia.org` has data
- Could cause missed lookups when the same site is accessed with/without `www.`

### Theory vs Reality

**Theory (what v0.6.0 promises):**
The domain knowledge system is described as "the self-improvement foundation" where "every navigation, block event, consent dismissal, and API discovery feeds back into future visits." The expectation is that repeat visits get faster, use the right wait strategy, start at the correct stealth tier, and skip consent dialogs automatically.

**Reality (what the code actually does):**
1. **Records correctly**: Visit counts, wait strategy averages, stealth tiers, consent selectors, API endpoints — all schema fields work and persist to disk
2. **Never reads back**: The loaded `domainRecord` on line 528 of `index.ts` is a dead variable. `adaptiveNavigate()` starts every visit fresh with the user-supplied `waitUntil` (default: `"load"`)
3. **Wait strategy is always caller-specified**: Even though the system knows Reddit needs `networkidle`, the next visit will try `load` first, hit an empty page, then adaptively escalate to `networkidle` again — wasting ~6 seconds every time
4. **Stealth tier resets**: If github.com escalated to stealth tier 2 in a past visit, `adaptiveNavigate()` still starts at Level 0 and re-escalates from scratch
5. **Consent selectors recorded but not replayed**: `recordConsent()` stores a selector, but there is no code that reads `consentSelector` on navigate and auto-clicks it

**What would actually improve token counts on revisits:**
The element counts cannot decrease because the same page always has the same DOM. The improvement path is:
1. **Use learned wait strategy**: Pass `domainRecord.waitStrategy.method` as the default `waitUntil` to `adaptiveNavigate()` instead of always starting at `"load"`. This avoids wasted retry cycles on SPAs.
2. **Use learned stealth tier**: Start at the known `stealthTier` instead of Level 0. Avoids wasted blocked-page detection and escalation on sites that always block.
3. **Replay consent selectors**: On navigate, if `consentSelector` is known, inject a post-nav click on it before returning the snapshot. This would reduce element counts by removing consent overlay elements.
4. **Learn page regions**: The `stableElements` array is empty for all domains. If populated with nav/footer selectors, these could be excluded from snapshots on revisit, genuinely reducing token output.

**Verdict:** The self-improvement system is an excellent data collection layer (Phase 1 of a feedback loop). It is NOT yet a closed feedback loop — the "act on knowledge" step (Phase 2) is unimplemented. The domain_knowledge tool works perfectly as an inspection/debugging interface.

### IDEA-009: Shared Sidecar — Multi-Terminal Browser Orchestration (v0.7.0)

**The vision:** Two Claude Code terminals, 8 browsers tiled across one screen, all coordinated. One agent researching competitors while another tests your app.

~~**Original design: shared sidecar server**~~ — REJECTED. Single point of failure, race conditions, zombie windows, port ownership drama, leader election complexity. Building a distributed system to arrange rectangles.

**Revised design: file-based coordination (no server)**

```
~/.leapfrog/tiles.json   ← shared grid state (all instances read/write)
~/.leapfrog/tiles.lock   ← file lock for atomic access
```

**How it works:**
- Each instance is fully autonomous — no server, no port, no leader
- On session create: acquire lock → read tiles.json → find open grid slot → claim it with my PID → write back → release lock → reflow my windows
- On session destroy: acquire lock → release slot → write back → release lock → reflow
- On tiles.json change (fs.watch): re-read → reflow my windows to match new grid
- Each instance only moves **its own windows** — reads full grid for position awareness, but never touches other instances' windows

**Crash recovery (built-in):**
- Before any write, scan all PIDs in tiles.json → `kill -0 pid` to check if alive
- Dead PID? Reclaim its slots automatically. No orphans, no manual cleanup.
- Surviving instances detect the freed slots on next read → reflow to fill the space

**Reflow behavior:**
```
Terminal 1 opens 3 browsers → tiles.json: [A1,A2,A3] → 3x1 grid
Terminal 2 opens 3 more     → tiles.json: [A1,A2,A3,B1,B2,B3] → 3x2, all 6 reflow
Terminal 1 crashes           → T2 detects dead PID → tiles.json: [B1,B2,B3] → 3x1, B's windows expand
Terminal 3 opens 2 more     → tiles.json: [B1,B2,B3,C1,C2] → 3x2, all 5 reflow
```

**Why this is better:**
- No server, no port, no SPOF, no EADDRINUSE
- Each instance is independent — if one dies, others don't care
- File locking is atomic and battle-tested (node `proper-lockfile` or `flock`)
- Works even if instances start minutes apart
- Crash recovery is automatic (dead PID detection)
- Zero network, zero discovery protocol
- Sidecar HTTP can still exist per-instance for its own `/screenshot/:id` etc. — just not for tiling coordination

### Self-Improvement Verdict

The domain knowledge system is **Phase 1 of a feedback loop** — it collects data correctly (wait strategies, stealth tiers, consent selectors, API endpoints, visit counts). But **Phase 2 is unimplemented**: the loaded data is never used to inform navigation decisions. Every visit starts fresh regardless of history.

Token counts (element counts) are perfectly stable across visits (157/157/157 for GitHub over 3 visits). This is expected — the same page has the same DOM. The real improvement path:
1. Use learned wait strategy → skip retry cascade → faster navigation
2. Use learned stealth tier → skip escalation → fewer blocked-page retries  
3. Replay consent selectors → auto-dismiss before snapshot → fewer DOM elements
4. Learn stable page regions → exclude from snapshots → actual token reduction
